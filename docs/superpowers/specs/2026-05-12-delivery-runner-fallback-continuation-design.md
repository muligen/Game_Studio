# Delivery Runner Fallback Continuation Design

## Background

The latest Delivery run stopped with many tasks still unexecuted because `qa: 准备测试计划文档` was marked failed after the agent returned fallback output. The QA agent did create project files under `F:\projs\GS_projects\proj_2c5d3a66`, including `docs/TEST_PLAN.md` and several test checklist documents, but the runner failed the task before recording filesystem changes. The DAG runner then stopped the entire plan on that single failed task.

This creates two bad outcomes:

- Real artifacts created before failure are invisible on the Delivery board.
- One failed or fallback task stops independent ready tasks that could safely continue.

## Goals

- Preserve and display artifacts generated by failed or fallback tasks.
- Allow independent tasks to continue after non-blocking task failures.
- Keep dependency safety: tasks that depend on a failed task must not run.
- Make failure reasons visible enough to debug: timeout, invalid JSON, Claude error, fallback output, or tool error.
- Prevent fallback checks from being treated as real quality evidence in acceptance.
- Provide an explicit recovery path so a failed task can be retried, retried with guidance, skipped when safe, or regenerated without losing history.

## Non-Goals

- Do not treat fallback output as successful execution.
- Do not skip the final acceptance gate.
- Do not change the Claude agent prompt strategy in this spec.
- Do not auto-ignore QA or quality failures; classify and surface them.

## Core Behavior

### Preserve Artifacts Before Failure Classification

The task runner must run filesystem diff after every agent invocation, regardless of whether the agent returned fallback output, invalid schema, or a failed result. If files changed, the task result records:

- `changed_files`
- `output_artifact_ids`
- `artifacts_preserved: true`
- a `file_changes_detected` task event

Only after artifact tracking should the runner classify the task outcome.

### Failure Classification

Introduce a task outcome classification used by task results, events, and board API responses:

- `hard_failed`: execution failed and downstream work must be blocked.
- `soft_failed_with_artifacts`: execution was not trustworthy, but useful artifacts were generated.
- `needs_validation`: artifacts exist but checks are missing or insufficient.
- `infra_failed`: Claude timeout, invalid JSON, permission error, tool error, or process crash.

Recommended default mapping:

- `dev`, `art`, `design` fallback: `hard_failed`
- `qa`, `quality` fallback with artifacts: `soft_failed_with_artifacts`
- any fallback without artifacts: `infra_failed`
- schema parse failure with artifacts: `soft_failed_with_artifacts` for QA/quality, otherwise `hard_failed`

### Retry And Attempt History

Delivery tasks should become attempt-based. A task keeps its stable `task_id`, while each execution creates a new attempt record:

- `attempt_number`
- `input_context_snapshot`
- `started_at`
- `completed_at`
- `runner_outcome`
- `failure_classification`
- `changed_files`
- `output_artifact_ids`
- `checks`
- `fallback_reason`
- `user_recovery_note`

Retry actions must not overwrite prior task results. They append a new attempt and mark previous artifacts as either `retained`, `superseded`, or `discarded`.

Supported recovery actions:

- `retry_same_task`: run the same task again with the original context and preserved dependency context. Use for timeout, transient Claude errors, invalid JSON, or tool interruptions.
- `retry_with_notes`: run the same task again with an operator note appended to task context. Use when partial artifacts exist and the agent should repair or complete them.
- `accept_artifacts_and_continue`: mark preserved artifacts as usable, while keeping a `needs_validation` warning so acceptance still requires independent proof.
- `skip_task`: mark a non-critical task as intentionally skipped. Only allowed when no hard dependency or acceptance criterion requires it.
- `regenerate_task`: create a replacement task or new attempt that supersedes the previous artifacts. Use when the task direction was wrong or the task split is flawed.

Retry attempts inherit upstream dependency results, preserved artifacts, and project files. The retry context must clearly state which files came from previous failed attempts.

### DAG Continuation Rules

The runner should no longer stop the whole plan on the first failed task.

For each ready batch:

- execute all ready tasks up to the pool concurrency limit.
- record each result independently.
- if a task fails, mark only tasks depending on it as blocked by failed dependency.
- continue with ready tasks that do not depend on failed tasks.

The runner stops only when:

- all runnable tasks are done, or
- remaining tasks are all blocked by failed dependencies, or
- the runner itself crashes.

### Dependency Blocking

When task A fails:

- direct dependents of A become `blocked_by_failed_dependency` or equivalent board-visible state.
- indirect dependents remain blocked and inherit the blocked reason through dependency context.
- independent tasks continue.

Each blocked task should record:

- `blocked_by_task_ids`
- `blocked_reason`
- `blocking_failure_classification`

### Plan Status

Plan status should reflect final runner state:

- `running_with_attention`: runner is still executing independent tasks while one or more tasks need review.
- `recovering`: a retry, regenerate, skip, or accept-artifacts action has been requested and the runner is resuming.
- `validating`: all tasks completed and acceptance is running.
- `accepted`: acceptance passed.
- `needs_attention`: at least one task failed or is blocked by failed dependency.
- `failed`: runner crashed or internal infrastructure failed before task state could be reconciled.

Avoid leaving the plan in `active` when there is no live runner and no runnable task can proceed.

### Human Intervention Points

Human intervention should be explicit and scoped to blocked or uncertain branches, not required for every task.

The runner should request attention only when:

- a hard-failed task blocks downstream work.
- a task produced artifacts but the result is not trustworthy.
- all remaining tasks are blocked by failed dependencies.
- acceptance fails after automatic repair attempts.
- the system cannot decide whether a task is safe to skip.

The operator can choose one of the recovery actions listed above. The selected action is stored as an auditable recovery decision with:

- `decision_id`
- `task_id`
- `action`
- `operator_note`
- `selected_artifact_policy`
- `created_at`

Human intervention should appear as a board action panel rather than a modal that blocks unrelated work. Independent runnable tasks continue unless the operator explicitly pauses the whole plan.

## Public Interface Changes

### TaskExecutionResult

Add optional fields:

- `failure_classification: str | None`
- `fallback_reason: str | None`
- `artifacts_preserved: bool = false`
- `blocks_downstream: bool = true`
- `blocked_by_task_ids: list[str] = []`
- `attempt_number: int`
- `supersedes_result_ids: list[str] = []`
- `artifact_policy: "retained" | "superseded" | "discarded" | None`

### Task Recovery Decision

Add a persisted recovery decision record or equivalent task event metadata:

- `id`
- `plan_id`
- `task_id`
- `action`
- `operator_note`
- `artifact_policy`
- `created_at`
- `created_by`

### Delivery Board API

Expose these fields per task when available:

- failure classification
- fallback reason
- preserved artifact paths
- blocked-by dependency ids
- whether the runner continued after the failure
- attempt history summary
- available recovery actions
- last recovery decision
- artifact policy for each failed attempt

### Task Events

Add or standardize events:

- `fallback_detected`
- `artifacts_preserved_after_failure`
- `task_soft_failed`
- `task_blocked_by_failed_dependency`
- `runner_continued_after_failure`
- `task_retry_requested`
- `task_retry_started`
- `task_retry_completed`
- `task_regenerated`
- `task_skipped_by_operator`
- `artifacts_accepted_by_operator`

## UI Requirements

The Delivery board should make stopped or partially stopped runs understandable at a glance:

- Agent Pool shows worker failures with original reason and classification.
- Task card shows `soft failed`, `hard failed`, or `blocked by dependency`.
- Failed task details include generated artifacts, if any.
- Plan banner summarizes why execution stopped or whether it continued:
  - "1 task soft-failed; 2 independent tasks continued."
  - "5 tasks blocked by failed dependency: task_abc."
- Failed task details show attempt history with input, duration, generated artifacts, checks, and error reason.
- Failed task details expose recovery actions:
  - Retry
  - Retry with notes
  - Accept artifacts and continue
  - Skip task
  - Regenerate task
- Recovery actions must preview their impact:
  - which downstream tasks will become runnable.
  - which artifacts will be retained or superseded.
  - whether final acceptance will still require extra validation.

## Acceptance Rules

Acceptance gate must not trust fallback-generated checks. If a task has `fallback_used`, `soft_failed_with_artifacts`, or `needs_validation`, the acceptance evaluator must require independent validation evidence such as command output, Playwright evidence, or explicit non-fallback QA/quality checks.

## Test Plan

### Unit Tests

- Fallback task with file changes records `changed_files`.
- QA fallback with artifacts becomes `soft_failed_with_artifacts`.
- Dev fallback becomes `hard_failed`.
- Fallback without artifacts becomes `infra_failed`.
- Documentation-only checks are not accepted as final validation evidence.
- Retry creates a new attempt and preserves previous attempt history.
- Retry with notes injects operator guidance into the next task context.
- Accepting artifacts marks them retained but keeps `needs_validation`.
- Skipping a required dependency is rejected.

### Runner Integration Tests

- Given three ready tasks where design succeeds, QA fallback writes files, and art succeeds, the runner completes design and art while marking QA soft-failed.
- Given dev depends on failed QA, dev is not executed and is marked blocked by failed dependency.
- Given independent ready tasks after a failure, runner continues them.
- Plan ends as `needs_attention`, not stale `active`, when blocked tasks remain.
- Retrying a failed task unblocks its dependents after the retry succeeds.
- Retrying with notes records the operator note and appends it to the agent context.
- Regenerating a task supersedes previous artifacts and updates downstream dependency context.
- Accepting preserved artifacts lets downstream tasks proceed but final acceptance requires independent validation evidence.

### Board API Tests

- Failed task response includes classification, fallback reason, preserved artifacts, and blocking flag.
- Blocked task response includes `blocked_by_task_ids`.
- Agent Pool error display includes failure classification and original message.
- Failed task response includes available recovery actions.
- Task response includes attempt history and artifact policy.
- Recovery decision API records operator note and selected action.

### Regression Scenario

Reproduce the `proj_2c5d3a66` case:

- QA writes `docs/TEST_PLAN.md` and test checklist files.
- QA returns fallback or invalid schema.
- Task result still records those files.
- Independent ready tasks continue.
- Dependent tasks remain blocked with a visible reason.

## Implementation Notes

- Move filesystem diff detection before fallback failure handling in `studio/runtime/graph.py`.
- Keep `service.fail_task()` for hard failures, but add a way to persist changed files and classification.
- Consider adding `complete_task_with_attention()` or `record_task_result()` to avoid overloading `done` and `failed`.
- Update dependency advancement logic to distinguish failed dependencies from merely unfinished dependencies.
- Persist the original `fallback_reason` from agent trace when available.
- Model recovery as a small explicit state machine, not as ad hoc task status edits.
- Keep attempt records immutable; new retries create new attempts.
- Treat operator recovery decisions as inputs to the runner so the DAG can resume deterministically.
